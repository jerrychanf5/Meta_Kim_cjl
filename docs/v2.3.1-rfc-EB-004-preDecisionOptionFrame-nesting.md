# RFC: EB-004 — `preDecisionOptionFrame` Nesting Normalization

**Status**: draft
**Target release**: v2.3.1
**Author**: main-thread (post-v2.3.0 carry-forward)
**Created**: 2026-05-26
**Related bug surface**: `canonical/runtime-assets/shared/hooks/spine-state.mjs`

## 1. Problem statement

The `choiceSurfaceState` field has **two valid locations** in the spine state, with the hook consuming only the top-level location. This violates PRIN-02 (single source of truth) and PRIN-03 (no hidden coupling).

### 1.1 First-hand evidence from v2.3.0

The v2.3.0 main-thread initially set `preDecisionOptionFrame.choiceSurfaceState = "completed"` inside the frame, observed in spine state lines 292-305 at that point:

```json
{
  "preDecisionOptionFrame": {
    "frameId": "v2.3.0-batch-fix-choice-card",
    "solutionChoiceState": "user_chose_path_A_terminal_scope_v2_3_0_minor",
    "userSelection": { ... },
    "choiceSurfaceState": "completed",    // <-- Set here first
    ...
  }
}
```

The hook continued to deny Bash with `"Execution cannot start before execution confirmation is completed or an explicit choiceGateSkip is recorded. Missing: choiceSurfaceState=completed."`.

Reading `canonical/runtime-assets/shared/hooks/spine-state.mjs:849`:

```javascript
const choiceState = normalizeChoiceSurfaceState(state.choiceSurfaceState);
```

The hook reads `state.choiceSurfaceState` **top-level only**. The frame-internal value was ignored. Main-thread had to add a second top-level field to satisfy the hook:

```json
{
  "currentStage": "execution",
  "choiceSurfaceState": "completed",     // <-- Hook reads only this
  "preDecisionOptionFrame": {
    "choiceSurfaceState": "completed",   // <-- Documentation only; no enforcement
    ...
  }
}
```

This duplication is confusing, error-prone, and undocumented in the spine-state schema.

### 1.2 Related ambiguity

The same pattern recurs for other fields described in the `meta-theory` skill's "Choice Surface Gate" section:

- `solutionChoiceState` — currently only inside frame
- `choiceGateSkip` — `hasChoiceGateSkip(state)` checks BOTH `state.choiceGateSkip` AND `frame.choiceGateSkip` AND `state.intentGatePacket?.choiceGateSkip` (line 832-836)

So three fields with related semantics live at three different ambiguous combinations of top-level/frame-nested. New contributors must read hook source to know which works.

## 2. Goals

- Single canonical location for each choice-surface field.
- Hook reads exactly one location per field.
- Schema validation rejects spurious duplicates with a clear error message.
- Backwards-compatible migration path for in-flight spine states.

## 3. Non-goals

- Do NOT change Choice Surface Gate semantics (when a confirmation is required vs not). EB-004 is purely a location normalization.
- Do NOT bundle with EB-002 (`read_only_verifier`) — independent RFCs to limit blast radius per Warden C5.

## 4. Proposed design

### 4.1 Canonical field locations (chosen direction: top-level)

| Field | New canonical location | Rationale |
|---|---|---|
| `choiceSurfaceState` | top-level (`state.choiceSurfaceState`) | already where hook reads |
| `solutionChoiceState` | top-level (`state.solutionChoiceState`) | promote out of frame; frame describes the question, state stores the answer |
| `choiceGateSkip` | top-level (`state.choiceGateSkip`) | drop the dual-lookup in `hasChoiceGateSkip` |
| `preDecisionOptionFrame` itself | top-level (`state.preDecisionOptionFrame`) | unchanged; it's a packet |
| Inside the frame | `frameId`, `questions[]`, `candidatePaths[]`, `recommendedDefault`, `questionCount`, `wardenReapprovalRequired`, `blastRadiusEstimate` (question definitions, not user answers) | unchanged |

### 4.2 Hook simplification

`spine-state.mjs::hasChoiceGateSkip` collapses to:

```javascript
function hasChoiceGateSkip(state) {
  return !!state?.choiceGateSkip;
}
```

(removes dual-lookup of frame + intentGatePacket).

### 4.3 Schema validator (NEW)

`scripts/validate-project.mjs` gains a check `validateSpineStateChoiceFieldLocations(state)`:

- FAIL when `state.preDecisionOptionFrame.choiceSurfaceState` is set (must be on top-level only).
- FAIL when `state.preDecisionOptionFrame.solutionChoiceState` is set (must be on top-level only).
- FAIL when `state.preDecisionOptionFrame.choiceGateSkip` is set (must be on top-level only).
- PASS for top-level usage.
- Error message must cite this RFC.

### 4.4 Backwards-compatible migration (deprecation period)

For one minor release (v2.3.1):

- Hook continues to read top-level (current behavior).
- Validator warns (does NOT fail) when frame-nested copies exist; warning message: `"Deprecated: 'preDecisionOptionFrame.choiceSurfaceState' will be removed in v2.4.0; move to top-level state.choiceSurfaceState"`.

In v2.4.0:

- Validator hardens to FAIL.

### 4.5 Migration helper script

`scripts/migrate-spine-state-eb004.mjs` (new):

- Read each spine state under `.meta-kim/state/*/spine/spine-state.json`.
- If frame-nested copies exist and top-level missing: promote.
- If both exist and disagree: log error + abort (manual intervention required).
- Idempotent (safe to re-run).

## 9. Why this is LOW severity

- v2.3.0 main-thread worked around the bug in one Edit call.
- No data loss possible; hook simply reads the wrong location and rejects.
- No production hook security implications.
